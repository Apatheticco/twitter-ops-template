# 长推文拆解器 Thread Builder

把研报、长文、分析报告拆成结构清晰、每条独立成立、风格一致的Twitter线程。

---

## 核心定位

```
已有长内容（研报/文章/白皮书）→ 【thread-builder（拆解转化）】→ 发布
```

**与 tweet-composer 的分工：**
- **tweet-composer Thread模式**：从 topic-engine 的角度**原创**写Thread（无原文）
- **thread-builder**：把**已有长内容**转化为Thread（有原文要拆解）

两者共享同一套风格规范和字数限制，但创作流程不同。

---

## 核心挑战

长文变Thread不是简单分段。难点在于：
1. 原文5000字，Thread最多10-15条推文，必须大量压缩
2. 每条推文要独立可读（有人只看到其中一条被转发的）
3. 第一条决定整个线程的阅读率
4. 风格要和你的人设一致，不是原文的学术/媒体风格

## 工作流

### 第一步：消化原文

接收用户提供的长内容（研报/文章/白皮书/推文串/笔记/语音转文字），做核心提取：

```
原文分析：
- 核心论点：[最多3个，一句话一个]
- 关键数据：[原文中最有冲击力的3-5个数据点]
- 支撑论据：[每个论点的主要论据]
- 作者隐含假设：[没明说但逻辑依赖的前提]
- 最薄弱环节：[最容易被反驳的地方]
- 你的独特角度：[你对这个话题能补充什么]
```

### 第二步：确定线程长度和结构

根据内容复杂度选择模板：

#### 🔹 短线程（3-5条）— 快速观点输出
适用于：新闻评论、单一观点展开

```
结构：
推文1：核心观点 + 为什么值得看（hook）
推文2：最强论据或最关键数据
推文3：反向风险或补充视角
推文4（可选）：历史对比或类比
推文5（可选）：结论 + 互动提问
```

#### 🔹 中线程（5-8条）— 深度分析
适用于：研报摘要、项目分析、市场复盘

```
结构：
推文1：一句话抛出最反直觉的结论（hook）
推文2：背景铺垫（30秒能理解的版本）
推文3-4：核心论证（数据+逻辑）
推文5-6：关键转折或反面论据
推文7：实操启示（对交易/投资的意义）
推文8：总结 + 你的独立判断 + 提问
```

#### 🔹 长线程（8-15条）— 系统性深度
适用于：白皮书解读、行业研究、宏观分析

```
结构：
推文1：颠覆性的开头（hook）
推文2：为什么这个话题重要（建立期待）
推文3-5：第一个核心论点（论据+数据）
推文6-8：第二个核心论点
推文9-11：第三个论点或反面分析
推文12-13：综合判断+实操意义
推文14：风险提示
推文15：总结 + 强有力的收尾 + CTA
```

### 第三步：逐条撰写

每条推文遵循以下规则：

**硬性规则：**
- 中文推文 ≤ 140汉字（Twitter CJK字符计2，280字符额度≈140汉字）
- 每条独立完整，被单独转发也能看懂
- 标注序号（1/8, 2/8...）
- 包含 $TICKER 或 #hashtag（线程整体2-3个，不是每条都加）

**风格规则：**
- 不照搬原文措辞，用你自己的交易员语气重写
- 强观点：在原文基础上形成你的判断，不是中立转述
- 数据具体化：原文说"大幅增长"你要说"增长了47%"
- 每3-4条插一个节奏变化（数据→观点→反问→数据→类比）

**第1条推文（决定生死）：**
必须做到以下至少一个：
- 抛出一个反直觉的结论
- 用一个惊人的数据开头
- 直接挑战主流观点
- 提出一个让人忍不住想知道答案的问题

**最后1条推文：**
- 总结核心观点（不超过2句）
- 给出你的独立判断（不是原文作者的判断）
- 以提问或CTA收尾

### 第四步：质量检查

```
线程检查清单：
- [ ] 推文1是否足够有吸引力（5秒决定阅读）
- [ ] 每条推文是否 ≤ 280字符
- [ ] 每条是否独立可读
- [ ] 核心数据是否都保留了
- [ ] 是否有你的独立判断（不是纯转述）
- [ ] 是否包含反向风险
- [ ] 没有价格预测
- [ ] 风格是否一致（交易员语气）
- [ ] 最后一条是否有CTA
- [ ] 总长度是否合理（不水也不赶）
```

### 第五步：输出

```
## 📝 线程方案

**原文来源**：[标题/链接]
**线程长度**：X 条推文
**核心论点**：[一句话]

---

🧵 1/X
[推文内容]

🧵 2/X
[推文内容]

...

🧵 X/X
[推文内容]

---

## 📊 线程规格
- 总字符：约 XXXX
- 核心数据点：X 个
- 预计阅读时间：X 分钟
- 建议发布时间：[基于内容类型]

## 💡 发布建议
- 发布节奏：[一次性发完 / 间隔X分钟]
- 推文1可单独作为独立推文发布（如果线程效果不确定）
- 可从推文[X]处截断成短线程版本
```

## 输入格式

接受任意长内容输入：
- 研报/文章链接（我会用WebSearch/WebFetch获取）
- 直接粘贴的长文本
- 白皮书或PDF内容
- 别人的推文串（整合+再创作）
- 语音转文字的粗糙笔记
- 要点列表/大纲

## 数据补充：按需回调 MCP

原文数据不足或需要更新时，回调 MCP 工具：

```
补充实时价格 → crypto_realtime_price_batch / realtime_price
补充链上数据 → open_channel_feeds / whale_trader_feeds
补充技术指标 → market_analyst(symbol, indicators="all")
补充宏观数据 → finance_tool_treasury_rates / fred_get_series
搜索相关报道 → search_finance_news / open_search_feed
获取原文（URL）→ WebFetch
```

优先使用原文已有数据，只在需要更新或补充时回调。

## 与上下游的衔接

### ← 输入来源
- 用户直接提供长内容（研报/文章/白皮书/笔记）
- trend-scout 发现的深度文章（通过 open_feed_articles / search_finance_news 获取）
- topic-engine 标注为"适合Thread"的话题（转交 thread-builder 做深度拆解）

### → 输出去向
- 发布前经过 tweet-composer 的发布检查清单
- 发布后对接 engagement-manager 追踪线程表现
- 高表现线程入库 content-vault

# 推文撰写 Tweet Composer

topic-engine 的下游。将选题角度转化为可直接发布的推文内容。

---

## 核心定位

```
topic-engine（选题+角度）→ 【tweet-composer（撰写）】→ 发布 → engagement-manager（互动管理）
```

tweet-composer 不做选题判断（topic-engine 已完成），专注于**将角度写成高质量推文**。

---

## 价格数据铁律

1. **单一信源**：推文中所有价格/涨跌幅，只能引用 `crypto_realtime_price_batch` 返回的数字。新闻标题、上游简报叙事区里的价格描述一律视为"历史叙述"，不作为当前数据使用。
2. **写作前价格二次确认**：如果推文涉及具体价格/涨跌幅，开始撰写前必须重新调一次 `crypto_realtime_price_batch`，用最新返回值写稿。不引用上游简报里的价格数字（从采集到写稿可能已过几分钟，价格会变）。
3. **质检价格核验**：发布前检查新增硬性项——推文中所有价格/数据必须与最近一次 API 返回值一致。

## 工作流程

### 第一步：接收输入

**标准输入（来自 topic-engine）：**
```
- 核心观点：[一句话]
- 数据支撑：[数据点列表]
- 开头金句：[推文第一句]
- 反向风险：[对立面]
- 发布形式：单推 / Thread / 引用推
- 角度模板：[模板编号+名称]
```

**自由输入（用户直接给）：**
- 任何话题/观点/新闻链接
- 此时 composer 自动判断最佳发布形式和风格

### 第二步：根据发布形式撰写

#### 📝 单推模式（Single Tweet）

**硬性限制**：
- Twitter 对中文字符计2，实际上限 ≈ **140个汉字**（或280英文字符）
- $TICKER 和 #Hashtag 计入字数
- 目标：120汉字以内留出标签空间

**结构**：
```
[开头金句/Hook — 1句，<40字]

[核心论证 — 1-2句，含数据]

[反向风险或结尾钩子 — 1句]

$TICKER #Hashtag ×2-3
```

**结尾钩子三选一**：
- 提问型：激发讨论（"你怎么看？"）
- 反转型：制造惊奇（"但你可能不知道…"）
- CTA型：引导互动（"同意的转发"）

#### 🧵 Thread 模式（3/5/7条）

**第1条（决定成败）**：
- 主观点 + 最强Hook
- 必须独立成立 — 很多人只看第1条
- 以 🧵 开头标记线程

**中间条（展开论证）**：
- 每条围绕一个论点/数据点
- 每条独立完整但逻辑递进
- 使用换行营造节奏感

**最后1条（收尾）**：
- 总结核心观点
- 反向风险/对立面
- 互动钩子（提问或CTA）

**格式**：每条标注 `[1/N]`，每条 ≤ 140汉字

**Thread 长度选择**：
| 内容量 | 推荐长度 | 适用场景 |
|--------|---------|---------|
| 1个观点+1组数据 | 3条 | 快速观点展开 |
| 1个观点+多组数据+对比 | 5条 | 标准深度分析 |
| 复杂叙事+历史对标+多维分析 | 7-10条 | 深度研报级Thread |

#### 💬 引用推模式（Quote Tweet）

**适用场景**：回应他人推文、对新闻链接做评论

**结构**：
```
[你的锐评 — 1-2句，有明确立场]
[补充数据或反向风险 — 1句]

$TICKER
```

**注意**：引用推的字数空间更小（被引用内容占据展示空间），控制在 **80汉字以内**。

### 第三步：风格适配

根据 topic-engine 指定的角度模板，自动匹配写作风格：

| 角度模板 | 写作风格 | 语气 |
|---------|---------|------|
| #1 直击对立 / #5 嘲讽 | 锐评版 — 犀利一针见血 | 强观点、戳痛点、允许温和脏话 |
| #2 数据讲故事 / #10 交易洞察 | 数据版 — 硬核数据说话 | 权威、精确、数字优先 |
| #7 突发快评 | 快讯版 — 速度第一 | 短平快、抢时效、1-2句定调 |
| #4 历史平行 / #6 新视角 / #8 教育 | 叙事版 — 讲故事 | Thread为主、逻辑递进 |
| #3 反向风险 / #11 宏观 | 分析版 — 理性平衡 | 数据+逻辑、必含对立面 |
| #9 对比分析 | 对比版 — 结构化对比 | 表格化、差异突出 |

### 第四步：发布前检查

**硬性检查（必须全部通过）：**

- [ ] **字数合规**：单推≤140汉字，Thread每条≤140汉字
- [ ] **事实准确**：数据、价格、事件可验证
- [ ] **价格核验**：推文中所有价格/涨跌幅与最近一次 `crypto_realtime_price_batch` 返回值一致
- [ ] **无价格预测**：不说"会涨到X" — 可以说"上涨动能更强"
- [ ] **有反向风险**：每条推文/Thread都包含对立面
- [ ] **有结尾钩子**：提问/反转/CTA三选一
- [ ] **$TICKER格式**：代币用$BTC不用#BTC
- [ ] **Hashtag 2-3个**：不多不少

**软性检查（尽量满足）：**

- [ ] 开头3秒有冲击力
- [ ] 语气符合交易员人设（参考 voice-guide.md）
- [ ] 短句为主，换行营造节奏
- [ ] 没有模棱两可的对冲语气

### 第五步：输出

**标准输出格式**：

```
━━━━━━━━━━━━━━━━━━━━━━━
📝 推文草稿 — [话题简称]
形式：单推 / Thread [N条] / 引用推
角度：[模板名称]
━━━━━━━━━━━━━━━━━━━━━━━

[推文正文]

━━━━━━━━━━━━━━━━━━━━━━━
✅ 发布检查
字数：[X/140] | 事实：✓ | 无预测：✓ | 反向风险：✓ | 钩子：✓ | 标签：✓

📊 预期效果
目标互动：[讨论型/传播型/教育型]
最佳发布时间：[建议时间+理由]
━━━━━━━━━━━━━━━━━━━━━━━
```

**如果用户要求多版本**，额外生成1-2个替代版本（不同风格），但标注推荐版本。

## 数据补充与价格确认

**价格二次确认（必须执行）**：
- 推文涉及具体价格 → 开写前调 `crypto_realtime_price_batch`，用最新值
- 不引用上游简报中的价格数字（已过时）

**其他数据补充（按需回调）**：
```
需要验证链上数据 → open_channel_feeds / whale_trader_feeds
需要KOL原文 → twitter_advanced_search（Agent子进程）
需要补充宏观数据 → finance_tool_treasury_rates / fred_get_series
```

非价格类数据优先使用 topic-engine 已提供的，避免重复调用。

## 语气 & 人设

统一引用 `/skills/twitter-ops/references/voice-guide.md`，核心要点：

- **强观点**：不模棱两可，有明确立场
- **简洁至上**：能一句话说的不说两句
- **直言不讳**：直接指出愚蠢做法
- **数据驱动**：有数据就上数据
- **不预测价格**：可以说趋势和逻辑
- **允许温和脏话**：偶尔用，但不人身攻击
- **语言**：中文为主，行业术语保留英文（BTC, ETH, DeFi, APY, MACD等）

## 绝对禁区

| 禁止 | 原因 |
|------|------|
| "BTC会涨到X万" | 价格预测，法律风险+容易打脸 |
| "你应该买/卖..." | 投资建议，合规风险 |
| "跟我交易保证赚" | 虚假承诺 |
| "用杠杆放大回报" | 引导借贷 |
| 人身攻击项目创始人 | 可以批评做法，不能人身 |
| 泄露个人资产具体数字 | 隐私+安全风险 |
